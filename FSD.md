# Functional Systems Design (FSD) — Модуль предзаказов

## 1. Пользовательские истории (User Stories)

### US_01: Оформление дефицитного товара с предоплатой
* **Как**: Авторизованный покупатель мобильного приложения «Континент».
* **Я хочу**: Оформить онлайн-предзаказ на товар с остатком «1 единица» со 100% предоплатой через банковский шлюз.
* **Чтобы**: Быть уверенным в том, что заказанный мною товар заблокирован под меня, будет гарантированно в наличии, и я смогу прийти и забрать его в любое удобное время в рамках лимита хранения.

## 2. Сценарии использования (Use Cases)

### UC_01: Оформление заказа с онлайн-предоплатой

**Предусловие (Pre-conditions):** Пользователь авторизован, дефицитный товар (остаток = 1) добавлен в корзину, нажата кнопка «Перейти к оплате».


| Шаг | Действующее лицо | Действие (Сценарий Happy Path) |
| :--- | :--- | :--- |
| 1 | Пользователь | Нажимает кнопку «Оплатить заказ». |
| 2 | Мобильное приложение | Переводит кнопку в статус `disabled`, включает лоадер и инициирует запрос `POST /api/v1/orders/hold`. |
| 3 | Бэкенд Аптеки | Проверяет наличие товара в БД. Накладывает временный резерв (товарный холд) на 15 минут. |
| 4 | Бэкенд Аптеки | Инициирует сессию в Платежном шлюзе (API Acquiring) и возвращает URL для оплаты. |
| 5 | Мобильное приложение | Выполняет редирект пользователя на защищенную страницу платежного шлюза. |
| 6 | Внешняя система | Банк принимает оплату и отправляет асинхронный Webhook (`POST`) на Бэкенд Аптеки. |
| 7 | Бэкенд Аптеки | Меняет статус заказа в БД на `RESERVED_PAID` (SCD тип 2) и продлевает резерв до 72 часов. |
| 8 | Бэкенд Аптеки | Публикует (push) событие `Order_Paid` в брокер сообщений Kafka для начисления баллов. |

**Постусловие (Post-conditions):** Товар заблокирован в БД, статус заказа изменен, событие отправлено в очередь Kafka.


## 3. Спецификация межсервисного взаимодействия (API Query Parameters)

### GET /api/v1/orders/status
Запрос от Сервиса Лояльности (Consumer) к Бэкенду сети «Континент» для проверки финального статуса заказа перед начислением бонусов.

* **Параметры запроса (Query String):**
  * `order_id` (INT): Уникальный идентификатор заказа.
  * `client` (STRING): Имя клиента. При передаче кириллицы строка подлежит обязательной трансформации через URL-Encoding.

* **Ожидаемый ответ (Response):** 
  * `200 OK` с телом бизнес-статуса заказа (JSON-объект, содержащий текущий статус в рамках SCD тип 2).

* **Логика обработки ответа (Business Logic):**
  * Если Бэкенд возвращает статус `DELIVERED` — Сервис Лояльности выполняет начисление бонусных баллов.
  * Если Бэкенд возвращает статус `CANCELLED` — начисление бонусных баллов жестко блокируется, транзакция лояльности закрывается со статусом `DROPPED`.
